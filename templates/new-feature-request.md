# New Feature Request

Use this template to request a new feature with enough context for a Staff Engineer-level implementation plan.

## Problem

Describe the user, business, or operational problem this feature should solve.

## Desired Outcome

Explain the measurable outcome or behavior that should exist when the feature is complete.

## Current Behavior

Describe how the system works today, including relevant entry points, workflows, APIs, UI states, jobs, events, or dependencies.

## Proposed Scope

List what should be included in this feature.

## Out Of Scope

List related work that should not be implemented as part of this feature.

## Acceptance Criteria

- The expected happy path is defined.
- Important error states are defined.
- Relevant edge cases are defined.
- Backward compatibility expectations are defined.
- Observability expectations are defined where applicable.

## Constraints

Document known constraints, such as deadlines, compatibility requirements, performance limits, external dependencies, compliance needs, or rollout requirements.

## Trade-Offs To Consider

- Simplicity vs flexibility.
- Short-term delivery vs long-term maintenance.
- Performance vs readability.
- Coupling vs duplication.
- Operational safety vs implementation speed.
- Backward compatibility vs cleaner design.

## Security And Reliability Notes

Call out sensitive data, authentication, authorization, webhooks, async processing, external integrations, data consistency, high-risk business workflows, or production risk.

## Test Expectations

List expected automated test coverage:

- Happy path.
- Error cases.
- Relevant business rules.
- Edge cases.
- Regression cases.

## Rollout And Rollback

Describe how this feature should be released, monitored, and rolled back if needed.

## Open Questions

List unknowns that should be answered before implementation starts.
